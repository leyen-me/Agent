# Agent Future Roadmap

## Current Assessment

当前这套 agent 已经不是单纯的提示词壳子，而是一个具备基础工程执行能力的系统，已经拥有这些核心能力：

- 任务规划与执行分离：`PlanAgent` 负责理解需求和拆任务，`ExecuteAgent` 负责落地执行。
- 工具化执行：支持列文件、搜索代码、读取文件、编辑文件、运行命令、启动后台服务、查看日志。
- 状态持久化：任务、请求、Plan 历史、后台任务都能落盘，具备续跑基础。
- 交互式终端：支持 `/help`、`/jobs`、`/job-log`、`/usage`、`/export` 等命令。

这意味着它已经可以承担单仓库内的大部分常见开发任务，例如：

- 阅读和定位代码
- 小到中等规模的代码修改
- 运行测试、构建、脚本
- 启动开发服务并查看日志
- 将复杂需求拆成多个可执行子任务

但它距离“稳定完成大多数真实工程任务”还差一层，主要差在可靠性和闭环，而不只是提示词。

## Core Principle

后续优化重点不应再只放在 prompt 上，而应放在下面这几个维度：

- 模型稳定性
- 工具设计和边界
- 自动验证闭环
- 上下文管理
- 评测体系
- 安全策略

一句话总结：

> 现在是“能做很多事”的阶段，后面要进入“稳定把事做成”的阶段。

## Priority Roadmap

### Phase 1: Raise Success Rate First

这一阶段目标是最快提升真实任务成功率。

1. 做上下文压缩和摘要机制
   - 当前对话消息会持续累积，长会话下会影响稳定性。
   - 需要在任务阶段或轮次阶段引入摘要，而不是无限堆原始消息。

2. 建立默认验证闭环
   - 修改后尽量自动执行 lint、test、build 或最小验证命令。
   - 如果验证失败，应让 agent 基于失败结果继续修复，而不是只报告失败。

3. 调整模型和采样策略
   - 编码任务优先稳定性，而不是发散性。
   - 需要根据模型能力重新评估 `temperature`、`top_p`、thinking 开关和默认模型。

4. 强化工具参数校验
   - 工具调用前做更严格的 schema 校验。
   - 对常见参数错误提供兜底修正或更清晰的错误信息。

### Phase 2: Improve Engineering Reliability

这一阶段目标是把 agent 从“能跑”提升到“更可控、更稳”。

1. 升级命令执行安全策略
   - 现在更偏 denylist。
   - 后续应逐步引入 allowlist、命令分类、目录限制、风险等级控制。

2. 升级文件编辑能力
   - 当前已有整文件写入、唯一块替换、按行替换，足够可用。
   - 后续可以增加 patch/diff 风格编辑，降低大文件修改风险。
   - 如果项目继续发展，可以考虑符号级或 AST 级编辑。

3. 增强失败恢复能力
   - 工具失败后要有更明确的重试策略。
   - 对超时、交互命令、日志未刷新、文件未命中等情况建立标准恢复路径。

4. 完善任务执行策略
   - 为不同任务类型定义默认流程，例如“改代码 -> 验证 -> 汇总”。
   - 对启动服务、查看日志、停止任务、继续执行形成更稳定的状态机。

### Phase 3: Build Long-Term Leverage

这一阶段目标是让 agent 能长期演进，而不是靠体感优化。

1. 建立 benchmark 和回归测试集
   - 设计一批固定任务，持续评估成功率。
   - 例如：修 bug、改接口、补测试、启动项目、定位报错、重构局部逻辑。

2. 建立可观测性指标
   - 记录任务成功率、平均轮数、工具失败率、验证通过率、上下文长度。
   - 用数据判断改动是否真的提升了 agent。

3. 做经验沉淀机制
   - 把高频失败模式、有效修复路径、项目默认约束沉淀成规则或模板。
   - 减少每次都靠模型临场发挥。

4. 逐步形成任务模板库
   - 比如“新增接口”“修复报错”“启动前端”“改配置”“写单测”。
   - 用模板缩短思考路径，提升稳定性。

## What To Improve Specifically

下面这些点最值得优先投入：

- 上下文裁剪、摘要、阶段记忆
- 修改后的自动验证和二次修复
- 更稳的模型配置
- 更严格的工具参数校验
- 更安全的命令执行策略
- 更可靠的编辑工具
- 更明确的失败恢复逻辑
- benchmark 与成功率统计

如果只能选前三个，建议顺序是：

1. 上下文管理
2. 自动验证闭环
3. benchmark / 评测体系

## The Role Of Prompting

后期不是只靠提示词。

Prompt 当然仍然重要，但它的作用会越来越偏向：

- 约束行为
- 说明流程
- 减少幻觉
- 引导工具使用顺序

真正决定上限的，更多是：

- 模型能力
- 工具是否好用
- 状态和记忆是否清晰
- 修改后是否会验证
- 失败后能否恢复
- 是否有评测闭环

可以这样理解：

- 前期：prompt 决定 agent 会不会乱做
- 中期：工具和任务编排决定 agent 能不能做
- 后期：验证、记忆、评测决定 agent 能不能稳定做好

## Suggested Execution Order

为了避免同时改太多导致系统变脆，建议按下面顺序推进：

1. 先做上下文摘要和历史压缩
2. 再做默认验证闭环
3. 然后调模型参数和默认策略
4. 接着补工具参数校验和错误恢复
5. 再升级命令安全和编辑能力
6. 最后建立 benchmark、统计和长期演进机制

## Success Standard

当下面这些指标明显改善时，说明 agent 进入了下一阶段：

- 同类任务的成功率稳定上升
- 长对话任务不会明显失控
- 修改后会主动验证，而不是停在“已修改”
- 遇到失败时能自动换策略，而不是直接卡住
- 后续优化更多依赖数据和 benchmark，而不是只靠主观感觉

## Final Conclusion

当前 agent 已经具备较完整的执行框架，已经能做不少事情。

接下来最重要的，不是继续把 prompt 写得更长，而是把它做成一个更稳定的工程系统。未来工作的核心方向应该是：

- 更稳的上下文管理
- 更强的验证闭环
- 更可靠的工具系统
- 更安全的执行边界
- 更明确的评测体系

后面的竞争力，主要来自工程化，而不是只来自提示词。
