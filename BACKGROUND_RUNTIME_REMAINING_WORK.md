# Background Runtime 剩余工作清单

## 1. 当前已完成到什么程度

结合 `BACKGROUND_RUNTIME_OPTIMIZATION_PLAN.md`，当前已经完成的部分大致如下：

### 阶段 A：基本完成

- 已补后台 runtime 相关内置测试
- 已修正 `README.md` 中与测试现状不一致的描述
- 已统一后台 job 的结构化字段与摘要
- 已覆盖 `ready` / `timeout` / `stop` / `process_exited` 等关键分支

### 阶段 B：基本完成

- `run_command` 已支持：
  - `background=true`
  - `wait_mode=none`
  - `wait_mode=exit`
  - `wait_mode=ready`
  - `ready_check`
- `start_background_service` 已改成兼容包装器，内部走统一的 `run_command(..., background=true, wait_mode="ready")`

### 阶段 C：已完成必要部分

- 后台 `ready` / `completed` / `stopped` 事件已写入 runtime
- 终端交互入口已能投递未读后台通知
- `ExecuteAgent` 写任务结果时，已可吸收当前任务相关的未读后台事件
- 后台事件已区分“终端已展示”与“任务结果已消费”两条消费轨迹

### 阶段 D：已完成必要部分

- `PlanAgent` 已更明确避免把“启动服务”拆成重复任务
- `ExecuteAgent` 已更明确：
  - `ready=true` 后默认视为主目标完成
  - `wait_mode=exit` 返回后不再额外轮询
  - `read_background_job_log` 不再作为默认二次确认步骤

## 2. 现在还没做的部分

下面这些是还没做，或者只做了前半步的内容。

### 2.1 事件驱动还不够深

当前状态：

- runtime 已经会产出后台事件
- 终端入口会展示事件
- `ExecuteAgent` 在写任务结果时也能消费事件
- 事件存储已区分终端展示消费与任务结果消费，旧 `delivered_at` 仍保留兼容
- `PlanAgent` / `ExecuteAgent` 后续轮次已能稳定拿到“未被任务结果消费”的后台事件摘要输入

还没做的：

- 让任务推进逻辑在某些场景下更偏“等事件驱动结果”，而不是仍主要依赖单次工具返回值
- 继续收紧“终端已展示但任务未消费 / 任务已消费但终端未展示”这两类事件在后续轮次中的解释策略

结论：

- 这一块的基础消费边界和后续轮次输入已经补齐，但还不算完全事件驱动

### 2.2 任务流与提示词还能继续收口

当前状态：

- 关键提示词已经明显收紧

还没做的：

- 进一步清理兼容期遗留心智，降低 `start_background_service` 在提示中的存在感
- 继续减少模型主动提及“后台作业”这类内部概念
- 继续压缩模型在后台任务上的“再读一次日志确认”倾向

结论：

- 这部分不是阻塞项，但还可以继续打磨体验

### 2.3 `start_background_service` 还处在兼容保留阶段

当前状态：

- 仍然保留为公开工具
- 但内部已是兼容包装器

还没做的：

- 评估什么时候可以降级其在 prompt 中的优先级
- 评估是否长期保留该工具，还是最终只保留统一的 `run_command`

结论：

- 现在保留是合理的，后续再决定是否进一步弱化

### 2.4 `read_background_job_log` 的最终定位还可以再收紧

当前状态：

- 语义上已经偏向“用户显式查询 / 失败定位”

还没做的：

- 是否需要把某些失败场景下的结构化错误再补强，进一步减少读日志次数
- 是否需要对日志读取增加更明确的“仅诊断用途”提示或摘要字段

结论：

- 不是当前必须项，但后续有优化空间

### 2.5 Monitor 能力完全没做

当前状态：

- 没有 `monitor_background_job`
- 没有持续流式观察能力
- 没有基于日志模式的订阅式通知

结论：

- 这是计划里的明确未完成项
- 但它也是当前最不紧急的一项

## 3. 如果只按“必要性”排序

如果后续时间有限，建议按下面顺序继续：

1. 进一步收紧事件消费边界
2. 继续弱化旧工具心智
3. 视实际需求决定是否做 monitor

## 4. 最值得继续做的下一步

如果下次继续，只建议先做这一项：

### 让任务推进更偏向事件驱动

目标：

- 让部分任务推进更依赖 runtime 事件，而不是额外补一次工具调用
- 进一步减少“看见事件后仍想再读一次日志确认”的模型倾向
- 明确事件对任务状态推进的影响何时可直接落到 `done` / `failed`

建议最小工作项：

1. 在任务推进链路里增加更明确的“事件已足够完成判断”规则
2. 对 ready / completed / stopped / timeout 等事件补充更面向任务状态的映射
3. 进一步减少模型因为“想再确认一次”而重复读取日志或重复解释同一事件

## 5. 不建议现在立刻做的

- 不建议现在为了 monitor 再扩工具面
- 不建议现在删除 `start_background_service`
- 不建议现在做大规模多文件重构
- 不建议现在引入复杂的后台事件总线或订阅系统

## 6. 当前状态一句话总结

当前后台 runtime 已经从“专用 service 工具集合”明显收敛到了“统一命令执行 + runtime 事件 + 兼容旧接口”的状态；剩下的主要是体验收口和事件消费边界打磨，不再是核心能力缺失。
