# Background Runtime 生产测试提示词建议

这份文档用于你后续在真实项目里手动测试当前已经实现的后台 runtime 能力。

## 1. 使用建议

测试时建议分三类环境：

- 一个前端项目：用于测 dev server、端口 ready、访问地址返回
- 一个 Python 或 Node 项目：用于测普通后台命令和退出通知
- 一个故意失败或不存在入口的项目：用于测失败分支和日志定位

建议每条提示词单独开一轮会话，避免前后任务状态互相污染。

## 2. 重点想验证什么

当前最值得验证的是下面这些行为：

- Agent 会不会优先走统一的 `run_command(background=true, wait_mode=...)`
- “启动服务”是否还会被拆成重复任务
- 服务 ready 后，是否会直接把任务视为完成
- 是否还会无意义地反复读取同一份后台日志
- 后台事件是否会自然体现在任务结果里
- 终端里是否会收到 runtime 通知

## 3. 启动服务类提示词

### 3.1 标准前端 dev server

```text
帮我启动这个项目的前端开发服务器，确认服务可用后告诉我访问地址。
```

期待现象：

- 会优先检查项目入口
- 不会把“再读日志确认一次”拆成单独任务
- 最终结果应直接给出服务已启动、是否 ready、访问地址

### 3.2 模糊启动请求

```text
把这个项目跑起来。
```

期待现象：

- 会先检查 README、package.json、pyproject.toml、Makefile 等
- 只有在无法安全判断时才追问
- 不应直接僵硬地暴露“我创建了一个后台作业”

### 3.3 预览服务场景

```text
启动这个项目的预览服务，确认能访问后告诉我地址，不需要持续盯日志。
```

期待现象：

- 应把“确认能访问”当成主目标
- ready 后直接结束该任务
- 不应继续重复读日志

## 4. 普通后台命令类提示词

### 4.1 后台执行并等待结束

```text
把这个项目的构建命令放到后台跑，等它结束后告诉我结果。
```

期待现象：

- 应走后台命令，而不是前台卡住
- 完成后应给出结束结果
- 不应自己加很多 sleep 轮询

### 4.2 后台执行但不等待

```text
把这个项目的长耗时命令放到后台执行，先不要等结果，只告诉我已经启动即可。
```

期待现象：

- 应更接近 `wait_mode=none`
- 返回应偏向“已启动”“可后续查看”
- 不应还在当前轮继续盯输出

### 4.3 watcher 场景

```text
启动这个项目的 watcher，让它在后台持续运行；只要确认它已经起来就行。
```

期待现象：

- 应把它当常驻后台进程处理
- 重点是“起来了”，不是持续追日志

## 5. 后台查询与管理类提示词

### 5.1 查询当前后台任务

```text
看看当前有哪些后台任务在跑。
```

期待现象：

- 应直接走只读查询
- 不应额外新建任务

### 5.2 查询某个后台任务日志

```text
读一下刚才那个后台任务的日志，帮我看有没有报错。
```

期待现象：

- 会走 `read_background_job_log`
- 适合用于失败定位或用户显式查询

### 5.3 停止后台任务

```text
把刚才启动的后台服务停掉。
```

期待现象：

- 会查询并停止对应后台任务
- 停止结果应自然表达，不应很机械

## 6. 失败与异常场景提示词

### 6.1 不存在明确启动入口

```text
启动这个项目，但如果你找不到可靠入口，就明确告诉我卡在哪一步。
```

期待现象：

- 会先检查常见入口
- 找不到时应明确说明依据
- 不应乱猜启动命令

### 6.2 ready 超时

```text
帮我启动这个服务；如果一段时间内没 ready，就直接告诉我，不用一直等。
```

期待现象：

- 应有超时边界
- 超时后应明确告诉你没有 ready
- 不应无界轮询

### 6.3 启动后立即退出

```text
启动这个服务；如果进程马上退出，直接告诉我退出了，并尽量给我定位线索。
```

期待现象：

- 应识别 `process_exited`
- 结果不应假装成功

## 7. 多轮连续测试提示词

这类更适合验证事件与任务结果是否衔接自然。

### 7.1 先启动，再查询

第一轮：

```text
启动这个项目的开发服务器，确认 ready 后告诉我地址。
```

第二轮：

```text
看看刚才那个后台服务现在还在不在。
```

第三轮：

```text
把它停掉。
```

期待现象：

- 上下文应连续
- 不应把每轮都当成全新项目

### 7.2 先后台构建，再问结果

第一轮：

```text
把构建命令放后台跑，先不用等。
```

第二轮：

```text
刚才那个后台构建结束了吗？
```

期待现象：

- 第二轮应优先查后台状态
- 不应重新发起相同构建

## 8. 专门验证“少重复日志轮询”的提示词

```text
启动这个项目的前端服务，确认 ready 就结束，不要为了保险反复读同一份日志。
```

```text
把这个长命令放后台跑，等它结束后直接告诉我结果，不要自己 sleep 轮询。
```

这两条主要是观察行为是否已经收敛。

## 9. 你测试时建议重点观察的信号

### 好现象

- 任务描述更完整但更少重复
- ready 后直接收口
- 结果里会自然出现后台服务 ready 或后台任务结束的信息
- 日志读取只出现在失败定位或你显式要求时

### 不理想现象

- 启动服务后又继续重复读日志
- 明明已经 ready 还额外拆任务
- 反复说“我创建了后台作业”
- 普通后台命令完成后仍旧自己 sleep 再查

## 10. 如果你时间很少，只测这 5 条

```text
帮我启动这个项目的前端开发服务器，确认服务可用后告诉我访问地址。
```

```text
把这个项目跑起来。
```

```text
把这个项目的构建命令放到后台跑，等它结束后告诉我结果。
```

```text
看看当前有哪些后台任务在跑。
```

```text
把刚才启动的后台服务停掉。
```

## 11. 一句话建议

如果你是回家后慢慢测，优先先测“启动服务”和“后台命令结束”这两条主链路；这两条能通过，说明这轮 background runtime 收敛基本是到位的。
