# Agent 项目要求

## 项目定位

- 这是一个以 `main.py` 为核心的单文件 Python Agent/CLI 项目。
- 项目的重点是稳定的任务规划、任务执行、工具调用和 prompt 约束，不要为了“更优雅”擅自改成多文件架构。

## 架构约束

- 保持单文件架构，除非用户明确要求，否则不要拆分模块、子包或大规模搬运代码。
- 新增能力时，优先在现有文件中通过函数、类、常量、数据结构整理实现，而不是额外创建新文件。
- 如果修改 `PlanAgent`、`ExecuteAgent`、tool 注册、任务流转、system prompt、runtime context 等核心链路，必须检查上下游是否仍然一致。
- 不要随意改动已有数据格式、持久化字段名、CLI 命令名、工具参数名；如果必须调整，应优先兼容旧行为。

## 路径与运行时目录

- 面向用户的工作区概念以 `WORKSPACE_DIR` 为准；涉及用户文件的路径解析、展示和命令执行目录，都应围绕它理解。
- 沿用当前目录约定：配置、任务、历史、临时文件等运行时数据放在 `.agent/`；日志放在 `.logs/`。
- 不要把运行时文件随意散落到工作区其他位置，也不要未经用户要求调整这些目录约定。

## Prompt 与 Agent 行为

- 项目要求尽量减少模型幻觉；新增 prompt 信息时，优先补充边界、权限、模式、上下文来源，不要堆砌无关实现细节。
- `runtime_context` 只放当前运行事实；项目规则、用户约束、开发规范应放在独立的指令层中。
- 面向用户的默认表达应简洁、直接、可执行；不要把内部实现细节或内部代理分工无意义地暴露给用户。

## 开发与修改原则

- 先阅读相关实现，再修改代码；避免凭猜测改动。
- 学习并延续现有代码风格、命名方式、注释风格和错误处理方式。
- 优先做小步、局部、可验证的修改；避免无关重构和顺手大改。
- 不要为了抽象而抽象；新增抽象层前，先判断是否真的能降低当前复杂度。

## 验证要求

- 只要改动是实质性的，完成后应做最小必要验证。
- 如果仓库里已有现成的测试、自测、lint 或运行检查，优先复用，不要发明新的重型验证流程。
- 如果无法验证，要明确说明原因和风险，不要假装已经验证通过。

## 输出与协作

- 默认使用简体中文与用户沟通，除非用户明确要求其他语言。
- 若用户的即时要求与本文件冲突，以用户当轮明确要求为准；否则应优先遵守本文件。
